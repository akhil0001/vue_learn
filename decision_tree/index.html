Java Message Service
The Java Message Service API is a Java Message Oriented Middleware API for sending messages between two or more clients. It is an implementation to handle the Producer-consumer problem. JMS is a part of the Java Platform, Enterprise Edition, and is defined by a specification developed under the Java Community Process as JSR 914.

Check if JMS has been properly configured and installed as per the documentation and it is up and running.

In most of the cases we have observed the below issue in the metrics.log file when there is issue related to Basic or Standard reports. Error: There is no corresponding session_key for this request in dw_key_session table, for this SID : 1503504807581-279d-2f5c-b11e

Is this error present in log file?

Yes No

Check if IST call has been suppressed from the application code in either pre-appinit or post-appinit. Check if kony.license.disableMetrics() API is called in the Visualizer application. This code should be removed if it exists.

Check middleware.log if all the params with name konyreportingparams are getting printed in the middleare.log file(enable DEUBG mode to middleware component. Refer http://docs.kony.com/konylibrary/konyfabric/kony_fabric_linux_install_guide/Default.htm#Change_Log_Levels.htm%3FTocPath%3DFAQs%2520and%2520Troubleshooting%7C_____7 for steps).

When the middleware received the correct payload then it will print the below line in DEBUG mode. “Kony reporting params are :” If the device details matches with the one printed then proceed further to check metrics.log.

Middleware will log the below messages if the device did not send the IST payload or processing fails. Search for the below logs in the middleware.log “Reporting params not found in the IST request”. Or “JSON Error” if the payload is not correct JSON or “Exception while logging session tracking information” if there is any error in the IST call from device to SDK

If the middleware.log is also correct check the DB if the data is getting pushed correctly. We have to check the konyreportsDB middleware_request and middleware_session table if there are any entries related to the device from which we are testing the application. If there is no data check if there are any entries in the invalid_requests table of reports DB. Collect the data/error message to debug further.
